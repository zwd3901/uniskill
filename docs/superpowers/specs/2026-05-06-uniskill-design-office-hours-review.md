## 问题描述

使用多个 AI 编码 Agent（CodeBuddy、Claude Code、自定义 Agent 等）的开发者，需要将同一个 Skill 文件复制到每个 Agent 的技能目录中。每次修改技能都需要重复复制粘贴，容易遗漏或版本不一致。

## 什么让它很酷

一个 `uniskill link` 命令，OS 级别的软链接，改一处，所有 Agent 自动生效。不需要理解每个 Agent 的技能加载机制，不需要手动复制。就像 `npm link` 但针对 AI Agent 技能。

## 约束

- 跨平台：Windows（含无需管理员权限的 junction）、macOS、Linux
- Node.js >= 18
- 零外部依赖思维负担（仅 3 个依赖：commander, js-yaml, chokidar）
- 软链接断开时需能检测（broken link 状态）
- 用户不应需要理解文件系统底层原理
- 作用域：仅处理目录级别的链接（每个 skill 是一个目录），不处理单个文件

## 前提（经用户确认）

P1: 软链接是同步技能文件的核心机制
P2: YAML 配置文件是最合适的配置格式
P3: TypeScript + Node.js 是合适的实现技术栈
P4: CLI 工具是最佳的交付形式
P5: 链接同步是最核心的价值主张
P6: 四个 CLI 命令覆盖 V1 核心场景 (init/link/unlink/status)，list 和 watch 推迟到 V2
P7: npm + GitHub 是最合适的分发渠道
P8: MIT License 适合这个项目

## YAML 配置 Schema

```yaml
# 中央技能目录，相对于配置文件路径或绝对路径
source: ./skills

# Agent 目标列表
targets:
  - name: codebuddy                # 目标名称（唯一标识符，用于 --target 过滤）
    path: ~/.codebuddy/skills      # Agent 技能目录路径（支持 ~ 展开）
    method: symlink                # symlink | junction | copy

  - name: claude
    path: ~/.claude/skills
    method: symlink
```

字段说明:
- `source`: 字符串，必填。中央技能目录路径，相对于配置文件或绝对路径
- `targets`: 数组，必填。至少一个目标
- `target[].name`: 字符串，必填，唯一。用于 `--target <name>` 过滤
- `target[].path`: 字符串，必填。目标路径，支持 `~` 展开
- `target[].method`: 枚举，必填。可选值: `symlink` | `junction`（仅 Windows）| `copy`

## CLI 命令规范

### `uniskill init`
- 功能: 在当前目录生成 `uniskill.yaml` 模板
- 行为: 检查 `uniskill.yaml` 是否已存在 → 若不存在则写入默认模板
- 选项: 无
- 用例: `uniskill init`

### `uniskill link`
- 功能: 根据配置文件为所有 target 创建链接（核心命令）
- 前提: `uniskill.yaml` 必须存在（`init` 和 `link` 是两个独立步骤，不自动 fallback）
- 选项:
  - `--target <name>`: 仅链接指定 target
  - `--dry-run`: 仅展示要执行的操作，不实际执行
- 行为:
  1. 校验 source 目录是否存在
  2. 检查 target 路径的父目录是否存在，不存在则创建
  3. 冲突检测：遍历每个 target，检查路径状态（参考错误处理策略中 ERROR 级别的描述）
  4. 根据 method 创建软链接 / junction / copy
  5. 输出每个 target 的操作结果
- 用例:
  ```
  uniskill link               # 链接所有 target
  uniskill link --target codebuddy  # 仅链接 codebuddy
  uniskill link --dry-run     # 模拟运行
  ```

### `uniskill unlink`
- 功能: 移除所有软链接（不删除源文件）
- 行为:
  - 对 symlink/junction: 删除目标路径的链接文件
  - 对 copy: 删除目标路径的目录（带 `--force` 确认）
  - 跳过不存在的靶点
- 选项: `--target <name>` 仅移除指定 target
- 用例: `uniskill unlink`

### `uniskill status`
- 功能: 检查每个 target 的链接状态
- 状态定义:
  - ✅ `linked` — 有效软链接或 junction，指向 source 存在且正确
  - ⚠️ `broken` — 链接存在但指向的 source 目录不存在
  - ➖ `missing` — 未创建链接
  - ❌ `conflict` — 目标位置存在目录或文件，且不是链接，或链接指向不同的 source
- 检测机制: 实时检查（每次运行 `status` 时遍历所有 targets），非缓存。使用 `fs.promises.readlink` + `fs.promises.stat`
- 用例: `uniskill status`

### `uniskill list`（V2）
- 功能: 列出 source 目录下所有子技能目录，及每个技能关联的 target
- 内部机制: 扫描 source 目录下的每个子目录 → 对每个 target 路径执行 `readlink` → 反向映射回 source 中的技能目录 → 建立关联关系表
- 状态: **推迟到 V2（详见未来版本规划）**

### `uniskill watch`（V2）
- 功能: 监听 source 目录变化，自动同步到所有 targets
- 行为:
  1. 启动时自动执行一次全量 link
  2. 使用 chokidar 监控 source 目录的 add/change/unlink 事件
  3. 每次变更后防抖 500ms，执行增量 link（仅更新受影响的 target）
  4. 启动前执行所有 targets 的健康检查（确认 links 有效），避免 chokidar 在 broken junction 上崩溃
- 选项:
  - `--daemon`: Windows 上使用前台运行方式（Windows 无原生 daemon），用户可用 `Start-Process -NoNewWindow` 自行后台化
- 状态: **推迟到 V2（详见未来版本规划）**

## 考虑的方案

### Approach A: 符号链接同步（选定）
- 总览: CLI 工具，通过 OS 级别软链接统一管理 skills 目录
- 工作量: S (3-5 天，含 Windows 测试)
- 风险: 低
- Pros: 最轻量；零侵入 Agent 配置；三平台原生支持；一次安装零心智负担
- Cons: 不解决技能版本管理；跨机器复制需要额外处理链接
- 复用: 直接利用 OS 文件系统能力

### Approach B: Git 仓库分发（registry 模式）
- 总览: 中央技能仓库，各 Agent 通过 git pull 分发
- 工作量: M (1 周)
- 风险: 中
- Pros: 版本管理免费；PR 贡献；版本锁定
- Cons: 需要 git 操作；对非 git 用户不友好

### Approach C: 包管理器分发（npm-based）
- 总览: 每个 Skill 作为独立 npm 包发布
- 工作量: M-L (1-2 周)
- 风险: 中-高
- Pros: 语义化版本；npm update 即升级
- Cons: 打包发布流程复杂；对非 npm 用户有门槛

## 推荐的方案

**Approach A** — 因为目标用户的真实痛点是"改一处，重复粘贴到三个地方"，而不是"管理技能版本"。A 以最小代价解决最痛的点。

## 错误处理策略

| 层级 | 场景 | 行为 |
|------|------|------|
| FATAL | 配置文件不存在 / 格式错误 | 输出错误信息并退出，exit code 1 |
| FATAL | source 目录不存在 | 输出错误信息并退出（link 命令） |
| ERROR | 创建链接权限不足 | 输出错误并跳过该 target，继续处理下一个 |
| ERROR | target 路径有冲突（文件/目录） | 输出冲突报告并跳过，要求用户手动处理 |
| WARN | symlink 已存在但指向不同的 source | 替换为新的链接（输出 WARN） |
| INFO | 所有操作正常完成 | 输出绿色 tick |
| DEBUG | 内部状态、chokidar 事件 | 通过 `--verbose` 标志开启 |

退出码：0 = 完全成功，1 = 至少一个 target 失败。

## 未来版本规划

### V2（发布后下一个迭代）

| 功能 | 优先级 | 说明 |
|------|--------|------|
| `uniskill list` | 中 | 列出 source 目录下各技能关联的 target。实现简单，但 V1 用户可通过 `ls ./skills` 手动查看 |
| `uniskill watch` | 高 | 文件自动同步。设计已定（chokidar + 500ms debounce + 增量 link），V2 优先补齐 |
| 增量 link 优化 | 中 | V1 的 link 是全量处理所有 targets，V2 可优化为仅处理受影响的 target（watch 模式需要此能力） |
| `--daemon` Windows 方案 | 低 | Windows 后台进程方案需进一步调研。watch 模式下随 watch 一起交付 |

### V3 候选

- `init --preset`: 自动检测已安装 Agent 并生成 targets
- npm registry 集成: 支持从中央仓库安装技能包（Approach C 的轻量版本）
- 图形化状态界面: 通过浏览器查看链接状态（类似 `status` 的 web 版本）

## 成功标准（V1）

- 用户在 4 条命令内完成配置和链接（npm i -g -> uniskill init -> uniskill link -> uniskill status）
- `uniskill status` 准确报告四种状态（linked / broken / missing / conflict）
- `uniskill unlink` 正确移除所有链接，不删除源文件
- 跨平台（Windows junction / macOS Linux symlink）路径正确，无权限错误
- 测试覆盖率 > 80%，含单元/集成/E2E

## 测试策略

| 层级 | 测试内容 | 工具 |
|------|---------|------|
| 单元测试 | linker 核心逻辑（跨平台 mock fs 操作） | vitest |
| 单元测试 | config 加载/校验/类型守卫/YAML 解析 | vitest |
| 集成测试 | 临时目录创建/删除链接链，验证四种状态 | vitest + tmp |
| E2E 测试 | CLI 命令全流程（init -> link -> status -> unlink） | vitest + execa |
| 平台测试 | Windows junction、Linux/macOS symlink 差异 | CI matrix |

边界情况覆盖：

| 边界情况 | 所属层级 | 说明 |
|---------|---------|------|
| 空 skills 目录 | 集成测试 | link 时 source 无内容应正常退出，status 应报 missing |
| 深嵌套路径 | 集成测试 | source/target 路径含 5+ 级子目录，链接创建应正确 |
| source 被删除后重建 | 集成测试 | link 后删除 source → status 报 broken → 重建 source → relink |
| target 路径含空格/中文 | 集成测试 | Windows/macOS 路径含空格和 Unicode 字符 |
| 权限不足（无写入权限） | 集成测试 | target 父目录只读时 link 应报 ERROR 并优雅跳过 |
| 重复执行 link | E2E 测试 | 已 linked 状态下再次 link 应显示 skip 而非报错 |
| --dry-run 不产生副作用 | 单元测试 | dry-run 应一直 report 但从不 write
